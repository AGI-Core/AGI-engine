# 三类受众怎么写

> 这份文档解释：如何在一份文档里同时服务 🐤 小白 / 🦾 后端 / ⚡ 老手 三类读者。
> Agent 写章节前先读这份，确保每一段都有清晰的受众策略。

## 三类受众速览

| 类型 | 已知 | 缺失 | 期待 |
| --- | --- | --- | --- |
| 🐤 A 类小白 | 编程基础、粗略听过项目名 | 领域概念、架构模式、术语 | 手把手带我走一遍 |
| 🦾 B 类后端 | Go/Python/Java 后端、常见架构模式 | 目标模块的具体领域 | 90 分钟内建立"这项目怎么做的"的心智模型 |
| ⚡ C 类老手 | 用过同类库、看过论文 | 这个项目的独有决策 | 30 分钟看完速查线，能面试用 |

---

## 🐤 A 类小白：怎么让他们跟上

### 痛点

- 一遇到术语就迷失（"什么是 RRF？为什么要 rerank？"）
- 一遇到"我们做了 X 优化"就断线（不知道原本会有什么问题）
- 一遇到跳步就掉队（"然后就写入 Milvus" —— Milvus 是啥？）

### 期待

- 每个生词先解释，再使用
- 每个决策先讲"没这个决策会怎样"，再讲"我们是怎么做的"
- 每一步都有源码定位，能自己去看

### 该点出的内容

✅ **主线里保留**：类比、一步步走、代码引用、最基础的 📦
✅ **主线里必须解释**：任何非通识术语（RRF / BM25 / cosine / kernel / MVCC ...）
✅ **主线里加"如果不这么做会怎样"**：让读者理解决策价值

### 该藏进 📦 的内容

📦 补背景：Embedding、TCP、TLS、KV 存储、事件驱动 ...
📦 补细节：算法公式、参数默认值来源、历史变迁

### 不要给他们的

❌ 复杂的伪代码 / 数学推导（藏 📦 或独立一章）
❌ "这个熟练开发者应该知道..."（假设失效就 5 秒钟内劝退）
❌ 一次性列出 20 个组件 —— 分批介绍

## 🦾 B 类后端：怎么让他们高效

### 痛点

- 写了 5 年后端，但不知道 <TOPIC> 领域的常识
- 不需要从"什么是 API"讲起，但要知道"为什么这里用 RPC 不用 HTTP"
- 时间紧，希望 90 分钟内能对项目有整体判断

### 期待

- 快速建立全景图（1 张 mermaid 大图 + 5 个组件表）
- 直击设计决策（"为什么用 X 不用 Y"）
- 代码引用能定位到关键行，不用他们自己找

### 该点出的内容

✅ **1 章要有 1 张大图**：架构图 / 数据流图
✅ **组件分工表**：一眼看清"谁负责什么"
✅ **设计决策专章**：把面试亮点集中放一处

### 该藏进 📦 的内容

📦 通用背景（Embedding / BM25 —— 后端可能不熟但 5 分钟能懂）
📦 边角优化（细节参数、rare code path）

### 不要给他们的

❌ 从"什么是 HTTP"讲起
❌ 大段假设读者是完全新手的絮叨

## ⚡ C 类老手：怎么让他们眼前一亮

### 痛点

- 用过 LangChain / LlamaIndex / vLLM 等同类库，很多概念自带
- 想看的是"这个项目跟别人不一样的地方在哪"
- 面试临阵磨枪，30 分钟要能"讲得出亮点"

### 期待

- 有 一份"设计决策"专章，对比"通用做法"和"本项目做法"
- 有 一份 面试 Q&A 附录，直接用
- 关键设计要有"面试加分点"提示

### 该点出的内容

✅ **04-design-decisions 章要展开**：每个决策按 5 段式（现象 → 类比 → 怎么做 → 不这样会怎样 → 一句话）
✅ **appendix-a-interview-qa 要给标准答案**：直接抄就能用
✅ **对比"通用做法"**：告诉他们"别的库怎么做的"

### 该藏进 📦 的内容

📦 基础通识（他们已经知道，不要打断阅读节奏）

### 不要给他们的

❌ 从"什么是向量检索"讲起
❌ 手把手 step by step（他们跳读）
❌ 藏起来的关键决策（明确说"这个是我们独特的做法"）

## 如何让三类共存

### 主线走小白线

主线（章节正文）按 🐤 小白线的节奏写：类比开场、每步都解释、术语先解释再用。

**理由**：主线覆盖 100% 读者。B/C 老手是"会跳读"的，A 小白是"跳一句就掉队"的——按最难的读者写，其他人自然舒服。

### 加 📦 补背景

📦 是给 🐤 A 类的救生衣。C 类老手看到 📦 会跳过，B 类可能扫一眼确认，A 类会认真读。

**理由**：📦 让文档"分层"。A 类跟不上就读 📦，B/C 类不需要就跳过。

### 加"设计决策 / 面试拆解"章

这两章是 ⚡ C 类的甜点。

- **04-design-decisions**：每个决策解释"通用做法 vs 本项目做法"
- **appendix-a-interview-qa**：三档 Q&A，拷打档专门给 C 类

**理由**：C 类老手主要来这里挖亮点，让他们直接找到就行。

### 三档阅读路径

在 README 里明确列：

| 读者 | 顺序 |
| --- | --- |
| 🐤 小白线 | 全读 |
| 🦾 后端线 | 00 → 01 → 内核章 → 04 → 附录 A |
| ⚡ 速查线 | 01（看图）→ 04 → 附录 A |

**理由**：让读者自己选路径，比"每章开头写'如果你是老手可以跳过...'"更清爽。

## 决策表：一段内容放主线还是 📦？

| 判断 | 放主线 | 放 📦 |
| --- | --- | --- |
| 是本章核心机制吗？ | ✅ | ❌ |
| A 类不看会不会掉队？ | ✅ | ❌ |
| C 类会觉得啰嗦吗？ | ❌ | ✅ |
| 是通用背景概念吗？ | ❌ | ✅ |
| 是历史 / 八卦吗？ | ❌ | ✅（若跟本章有关）/ 直接删（若无关）|

**当拿捏不准时的默认策略**：先放主线，写完自查发现 A 类跟不上就抽出来放 📦。**宁可主线略啰嗦，不要有断线的读者。**
